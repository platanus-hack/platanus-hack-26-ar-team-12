<objective>
Add the instrumentation and demo proof needed to show Phase 2 satisfies the roadmap: tap-to-transcript under 3 seconds, no LLM/network path, strict WhatsApp prefill, and warm one-sentence TTS feedback.
</objective>

<user_story>
As the demo team, we want a repeatable proof checklist for Plan C, so that the live WhatsApp slice can be rehearsed and verified before moving to Phase 3.
</user_story>

<must_haves>
- D-04: Tap -> final transcript latency is logged and manually verifiable under 3 seconds.
- D-11: Success remains intent launch accepted, not Accessibility foreground proof.
- D-12: Failure stops warmly without future-phase fallback.
- D-17: Android native TTS only.
- Phase 2 success criterion 2: demo command works with mode airplane activated as far as native STT setup permits on the demo phone.
</must_haves>

<threat_model>
## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| False confidence from unmeasured latency | medium | Log tap, recognizer result, and elapsed ms with stable `Beto-STT` markers. |
| Hidden LLM/network dependency enters Plan C | high | Add explicit grep/manual checks for absence of LLM calls in Plan C controller/matcher path. |
| Demo phone not seeded correctly | high | `02-DEMO-CHECK.md` lists WhatsApp package, contact alias, phone number format, TTS voice, and STT offline rehearsal. |
| Logs expose real phone number in demo artifact | medium | Demo checklist uses placeholders; implementation logs sanitized/digits-only URI only in debug. |
</threat_model>
