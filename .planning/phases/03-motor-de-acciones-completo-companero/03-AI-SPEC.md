## 1. System Classification

**System Type:** Hybrid conversational assistant + tool-calling action router

**Description:**
Beto receives Spanish-Argentina voice/text input from an older adult, decides whether the user is asking for a phone action or conversational support, and either routes to a fixed Android Intent tool (`send_whatsapp`, `make_call`, `send_sms`, `open_maps`) or answers in a warm Companion chat. Good behavior means the user does not need to understand internal modes: Beto either answers calmly or confirms and performs the intended phone action without substituting contacts, leaking sensitive data, or creating a long UI burden.

**Critical Failure Modes:**
1. Calls, SMS, Maps, or WhatsApp route to the wrong person, destination, app, or message.
2. Sensitive data such as DNI, phone numbers, or card numbers leaves the device unredacted in a Gemini request or debug log.
3. Gemini returns malformed/unknown tool output and the app crashes, hangs, or silently executes an unsafe fallback.
4. Companion responses become confusing, robotic, too verbose, or mode-aware in a way that burdens older adults.
5. Agentic fallback is accidentally enabled in Phase 3 and attempts screen operation before the Phase 4 safety loop exists.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Assistive mobile AI / elder digital inclusion

**User Population:** Older adults using an Android demo phone with a floating Beto bubble, voice-first interaction, native STT/TTS, and pre-seeded demo contacts.

**Stakes Level:** High for wrong actions and privacy; Medium for conversational tone. This is a hackathon demo, but the product promise is operating a phone on behalf of a potentially vulnerable user.

**Output Consequence:** Tool outputs trigger real Android Intents for calls, SMS, WhatsApp, or Maps. Chat outputs influence trust, comfort, and whether an older adult feels capable instead of confused.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|-----------------------|--------------------|--------|--------|
| Intent safety | Beto performs exactly the requested action or asks before acting. | Beto guesses, substitutes a contact, or acts without confirmation on ambiguous input. | High | Phase 3 CONTEXT D-04..D-07 |
| Simplicity for elders | The user hears/reads one clear next step in simple Argentine Spanish. | The user sees technical mode names, JSON/tool language, or multi-step instructions. | High | PROJECT.md, CLAUDE.md |
| Privacy trust | Sensitive identifiers are redacted before cloud calls and redacted payloads are visible in `Beto-LLM` logs. | Raw DNI, phone, or card values appear in outgoing payloads or logs. | High | REQUIREMENTS `PRIV-01`, `PRIV-02` |
| Demo robustness | Demo-critical commands route locally before Gemini and complete under latency budget. | The demo depends on cloud success for "llama a mi hijo" / "sms a Ana" / "mapa farmacia". | High | Phase 3 CONTEXT D-01, PITFALLS.md |
| Companion warmth | Replies are Argentine, calm, patient, useful, and not boring. | Replies are cold, patronizing, verbose, or generic chatbot filler. | Medium | Phase 3 CONTEXT D-14 |

### Known Failure Modes in This Domain

- Older adult says a relationship name ("mi hijo") and the resolver silently chooses a different contact when lookup fails.
- Gemini returns natural language instead of a tool call or invents a tool name under time pressure.
- A sanitization regex misses Argentine phone/DNI formats, undermining the privacy story during the pitch.
- Companion starts as a friendly chat but exposes internal routing language ("switching to action mode") and increases cognitive load.
- Network loss makes arbitrary Gemini-routed commands fail; demo-critical commands must remain deterministic-first.

### Regulatory / Compliance Context

No formal healthcare/financial regulation is in scope for the hackathon MVP. Privacy is still a product-critical constraint: do not transmit or persist raw DNI, phone, or card numbers unnecessarily; do not persist Companion history in Phase 3; do not add personal memory until a future consent/storage design exists.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Product/demo owner | Approves that flows match the pitch narrative and the adult does not need to understand modes. |
| Spanish-Argentina reviewer | Labels tone as warm, simple, correct, calm, and not boring. |
| Android engineer | Verifies Intents, Contacts resolution, logs, sanitizer placement, and no Phase 3 agentic execution. |
| Privacy reviewer | Checks redaction examples and confirms no sensitive payload leaves unredacted. |

## 2. Framework Decision

**Selected Framework:** Firebase AI Logic Android SDK (`com.google.firebase:firebase-ai`) with Gemini function calling

**Version:** Use Firebase Android BoM already established in the Android app; update to the latest compatible BoM only if the existing project build allows it. Existing research pins `firebase-bom:33.5.1`; current Firebase docs recommend BoM-managed `firebase-ai` rather than hardcoding the library version.

**Rationale:**
This is a native Android hackathon app. Firebase AI Logic is the first-party client SDK path for calling Gemini from Android without building a backend proxy, which saves critical sprint time and avoids embedding a Gemini API key directly in the codebase. Gemini 2.5 Flash is already selected in project research for low-latency tool calling, while Flash-Lite is acceptable for the lower-stakes Companion chat. The app should not use a Python orchestration framework (LangGraph/CrewAI/LlamaIndex) because the AI surface is a client-side Android tool router plus chat, not a server-side multi-agent workflow.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Google ADK | Good for Google/Gemini multi-agent apps, but adds server/agent framework overhead; Phase 3 only needs direct Android SDK calls and local dispatch. |
| LangGraph | Strong stateful workflow tool, but overkill for deterministic-first Android Intents and session-only Companion chat. |
| CrewAI | Python-only multi-agent abstraction does not fit native Android client execution. |
| OpenAI Agents SDK | Strong agent framework, but project stack and key-management decision favor Gemini/Firebase. |
| Anthropic Java SDK | Kept as fallback if Gemini tool-calling quality fails; requires API key/proxy tradeoff and adds setup time. |

**Vendor Lock-In Accepted:** Partial. Phase 3 accepts Gemini/Firebase for demo speed and Android integration. Keep `LlmClient` interface boundaries so a later fallback (`AnthropicLlmClient`) can be added without changing `ActionDispatcher`.

### Entry Point Pattern
```kotlin
private val strictJson = Json {
    ignoreUnknownKeys = false
    explicitNulls = false
}

val makeCallTool = FunctionDeclaration(
    name = "make_call",
    description = "Llama a un contacto resuelto. No usar si el contacto es ambiguo.",
    parameters = mapOf(
        "contact" to Schema.String(description = "Nombre o alias del contacto, por ejemplo 'hijo' o 'Pedro'.")
    )
)

val actionModel = Firebase.ai(backend = GenerativeBackend.googleAI())
    .generativeModel(
        modelName = "gemini-2.5-flash",
        tools = listOf(Tool.functionDeclarations(listOf(
            sendWhatsappTool,
            makeCallTool,
            sendSmsTool,
            openMapsTool
        ))),
        systemInstruction = actionSystemInstruction
    )

suspend fun routeAction(originalText: String): ActionDecision {
    deterministicMatcher.match(originalText)?.let { return it }

    val sanitized = sanitizer.redact(originalText)
    val response = actionModel.generateContent(sanitized.text)
    return parseAndValidateToolCall(response)
        .getOrElse { repairOnceThenSafeFallback(originalText, it) }
}
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `GenerativeModel` | Firebase AI Logic model client. | Create separate model instances for action routing and Companion chat. |
| Function declarations | Tool schemas exposed to Gemini. | Define `send_whatsapp`, `make_call`, `send_sms`, `open_maps`; do not expose `agentic_perform_action` in the Phase 3 allow-list. |
| System instructions | Model-level behavior rules. | Keep Motor de Acciones deterministic/structured and Companion warm/conversational. |
| Structured output / schema validation | Constrains or validates machine-readable responses. | Use Kotlinx serialization and strict schemas; retry once on malformed output. |
| Firebase App Check | Client abuse protection for Firebase-backed API access. | Optional for hackathon if time is tight, but preferred before any public release. |

### Common Pitfalls

1. Providing `agentic_perform_action` in the active tool list before Phase 4; this violates scope and can trigger unsafe screen operation.
2. Accepting unknown fields or unknown tool names; `ignoreUnknownKeys` must stay `false` for tool-call argument parsing.
3. Letting fallback change entities. Fallback is only a routing fallback, never a semantic substitution.
4. Using one prompt/model config for both action routing and Companion; they have different temperature, tone, and safety needs.
5. Logging raw user text after sanitizer was supposed to run. Log redacted payloads for proof, not raw sensitive strings.

## 4. Implementation Guidance

**Model Configuration:**
- Motor de Acciones: `gemini-2.5-flash`, `temperature = 0`, low `maxOutputTokens`, system instruction in Spanish, tool descriptions in Spanish, allow-list: `send_whatsapp`, `make_call`, `send_sms`, `open_maps`.
- Companion: `gemini-2.5-flash-lite`, moderate temperature around `0.4`, session-only chat history, system instruction for Argentine tone: simple, correct vocabulary, patient, calm, conversational, not boring.
- Model-name upgrade note: current Firebase docs list newer Gemini 3 preview models, but Phase 3 should avoid preview churn unless the team explicitly smoke-tests and updates the plan. `gemini-2.5-flash` remains the stable locked choice from project research.

**Core Pattern:**
1. Receive transcript from `AgentBus`.
2. Run `DeterministicMatcher` first for demo-critical phrases.
3. If no deterministic match, sanitize text and call Gemini action router.
4. Strictly parse tool name and arguments.
5. Resolve contacts with `DemoContacts.kt` first, Android Contacts second.
6. Confirm before executing actions that originate from Companion.
7. Dispatch only through fixed tools; reject `agentic_perform_action`.

**Tool Use:**
- `send_whatsapp(contact, message)` prepares WhatsApp according to Phase 2 decisions.
- `make_call(contact)` starts a call only after contact resolution succeeds.
- `send_sms(contact, message)` prepares or sends according to the future Phase 3 plan decision; if unspecified, require confirmation.
- `open_maps(query)` opens Maps for the requested destination.
- No arbitrary app/screen tool in Phase 3.

**State Management:**
- Motor de Acciones: stateless per command except deterministic cache by original input hash for retry/resilience.
- Companion: `ViewModel` stores current sheet messages only; discard on close.
- No persistent personal memory in Phase 3.
- Logs may store redacted payloads and tool results, never raw sensitive values.

**Context Window Strategy:**
- Keep prompts compact: system rules, 4 tool schemas, 3-5 few-shot examples, and the sanitized current utterance.
- Companion history should be short and session-only. If history grows, keep latest turns and a short non-sensitive summary in memory only.
- Do not send Android contact lists wholesale to Gemini. Resolve contacts locally and pass only candidate labels needed for clarification.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with Pydantic

The Android implementation uses Kotlinx serialization, but the conceptual validation model should mirror this Pydantic contract for eval fixtures and documentation:

```python
from typing import Literal, Optional
from pydantic import BaseModel, Field, model_validator

ToolName = Literal["send_whatsapp", "make_call", "send_sms", "open_maps"]

class BetoToolCall(BaseModel):
    tool: ToolName
    contact: Optional[str] = Field(default=None, min_length=1)
    message: Optional[str] = Field(default=None, min_length=1)
    query: Optional[str] = Field(default=None, min_length=1)

    @model_validator(mode="after")
    def required_args_match_tool(self):
        if self.tool in {"make_call"} and not self.contact:
            raise ValueError("make_call requires contact")
        if self.tool in {"send_whatsapp", "send_sms"} and (not self.contact or not self.message):
            raise ValueError("message tools require contact and message")
        if self.tool == "open_maps" and not self.query:
            raise ValueError("open_maps requires query")
        return self
```

Kotlin equivalent must use `@Serializable` data classes plus `Json { ignoreUnknownKeys = false }`. On validation failure: one repair retry, then safe fallback only if the original text still matches the same semantics.

### Async-First Design

Use `suspend` APIs and coroutines for Gemini calls. Never block the main thread while waiting for model output, Contacts queries, or STT/TTS coordination. UI state should move through explicit loading/listening/thinking/speaking/error events on `AgentBus`.

### Prompt Engineering Discipline

- Keep system instructions separate for action routing and Companion.
- Motor prompt must say: choose only allowed tools, never substitute contacts, ask/fail on ambiguity, do not expose internal mode names.
- Companion prompt must say: Argentine tone, simple but correct vocabulary, patient, calm, conversational, not boring, can route to actions with confirmation.
- Tool descriptions should be in Spanish to match user utterances and reduce bilingual drift.

### Context Window Management

Do not include full prior chat, full contact book, or full planning docs in runtime prompts. Use only:
- current sanitized utterance,
- short Companion session history,
- a few examples,
- local resolver result/candidate list when clarification is needed.

### Cost and Latency Budget

- Demo-critical commands should avoid Gemini entirely via deterministic-first routing.
- Gemini action routing target: p95 under 2.5s on demo hotspot; timeout around 8-10s with warm failure.
- Companion first response target: p95 under 4s; pre-warm model if practical.
- Cache successful action-routing outputs by normalized sanitized input hash for demo resilience, but never cache raw sensitive strings.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| Tool selection correctness | Pass: selected tool exactly matches intent. Fail: wrong tool, missing tool, or natural language when tool is needed. | Code + reference dataset | Critical |
| Entity/contact safety | Pass: contact/destination/message preserved or clarification requested. Fail: any substitution such as Pedro -> hijo. | Code + human review | Critical |
| Sanitization coverage | Pass: DNI, AR phone, and card examples redacted before model call and logs show redacted payload. Fail: raw sensitive value appears in outgoing payload/log. | Code tests + log inspection | Critical |
| Malformed tool recovery | Pass: one repair retry then safe fallback/fail. Fail: crash, infinite retry, unsafe fallback. | Integration tests | High |
| Companion tone | 1-5 rubric: 5 = Argentine, simple, correct, patient, useful, not boring; 1 = cold, verbose, technical, patronizing. | Human review; optional LLM judge calibrated against user/team labels | High |
| Latency for voice action | Pass: deterministic demo commands under 5s action start; Gemini-routed commands fail warmly on timeout. | Instrumented smoke tests | High |
| Scope enforcement | Pass: `agentic_perform_action` rejected in Phase 3. Fail: dispatcher runs agentic screen operation. | Code tests | Critical |

### Reference Dataset

**Size:** Start with 18 examples.

**Composition:**
- 6 deterministic demo phrases: call hijo, SMS Ana, Maps farmacia, WhatsApp variants.
- 4 arbitrary contact phrases: Pedro known, Pedro unknown, multiple Ana contacts, relationship alias.
- 3 privacy phrases: DNI, phone, card in command text.
- 3 malformed/edge model outputs: unknown tool, missing required field, extra field.
- 2 Companion prompts: emotional support and action request inside Companion.

**Labeling:**
Product/demo owner labels expected behavior and acceptable tone; Android engineer labels tool/action correctness; privacy reviewer labels redaction. LLM judges are optional and must be calibrated against these labels before being trusted.

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Tool allow-list | Gemini returns anything outside `send_whatsapp`, `make_call`, `send_sms`, `open_maps`. | Block, log, warm failure or safe deterministic fallback if same semantics. |
| Contact ambiguity | Contact resolver returns zero or multiple matches. | Ask one short clarification or fail warmly; never guess. |
| Sanitizer preflight | Outgoing text contains unredacted DNI/phone/card pattern after sanitizer. | Block Gemini call, log redaction failure without raw value, warm failure. |
| Agentic disabled | Any Phase 3 path requests `agentic_perform_action`. | Reject and log; do not execute. |
| Timeout | Gemini action routing exceeds timeout. | Cancel, fallback to deterministic if same semantics, otherwise warm retry prompt. |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| Tool accuracy | Run full 18-example fixture after prompt/schema changes. | Rework prompt/schema; block plan completion if critical fixtures fail. |
| Redaction coverage | Run regex tests on DNI/phone/card fixtures. | Add regex/test cases before planning execution proceeds. |
| Companion tone score | Team labels 5-10 sample responses. | Adjust system instruction and examples. |
| Latency | Smoke-test on demo phone/hotspot. | Increase deterministic coverage or pre-warm model. |

## 7. Production Monitoring

**Tracing Tool:** Structured Timber/logcat tags for hackathon: `Beto-LLM`, `Beto-Action`, `Beto-Intent`, `Beto-STT`, `Beto-TTS`. Arize Phoenix or Langfuse can be revisited post-hackathon.

**Key Metrics to Track:**
- Tool-call parse success rate.
- Tool selection accuracy on fixture set.
- Redaction pass/fail count.
- Contact ambiguity/failure count.
- p95 action-routing latency.
- Companion tone approval score during team review.

**Alert Thresholds:**
- Any redaction failure blocks demo readiness.
- Any entity substitution failure blocks demo readiness.
- Any Phase 3 agentic execution path blocks demo readiness.
- p95 demo-critical action start > 5s requires deterministic routing expansion.

**Smart Sampling Strategy:**
Review interactions where the user repeats the command, contact resolution is ambiguous, Gemini repair retry is used, sanitizer redacts content, timeout occurs, or Companion routes to an action. These are the high-signal places where the system can lose trust.
