## 1. System Classification

**System Type:** Hybrid Autonomous Agent

**Description:**
Beto's Phase 4 AI system is a constrained fallback agent for an Android app for older adults. When the deterministic/Intent path for WhatsApp clearly fails, the system serializes the current Accessibility tree, asks Gemini for a short internal plan plus exactly one executable UI action, executes that action through `AccessibilityService.performAction()`, and repeats under strict limits until the WhatsApp contact/message fallback succeeds or aborts warmly. "Good" means the agent helps recover one controlled WhatsApp scenario without becoming the primary path, without auto-sending messages, without looping, and without exposing technical errors to the user.

**Critical Failure Modes:**
1. The fallback becomes the main execution path instead of only running after a clear WhatsApp failure.
2. The model chooses unsafe or out-of-scope actions, such as sending the message, operating calls/SMS/Maps, or interacting with non-WhatsApp apps.
3. The loop repeats actions or stalls instead of aborting within 5 iterations / 15 seconds.
4. The model produces malformed action output that cannot be safely mapped back to a registered `AccessibilityNodeInfo`.
5. Beto speaks confusing, verbose, or technical copy to an older adult during fallback/failure.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Accessibility / assistive mobile automation for older adults

**User Population:** Older adults using a preconfigured Android demo phone, with possible low vision, low confidence with apps, and low tolerance for ambiguous technical states.

**Stakes Level:** High

**Output Consequence:** The model's action is executed on the user's phone UI through Accessibility APIs. Even in the demo-only scope, a wrong action can confuse the user, type into the wrong field, send unintended content if not guarded, or leave the app in an unclear state.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|------------------------|--------------------|--------|--------|
| Bounded autonomy | Fallback runs only after clear WhatsApp failure and stays inside WhatsApp/contact search. | Agent starts automation for normal successful Intents or tries unrelated apps. | High | `.planning/phases/04-loop-ag-ntico-de-respaldo-ux-senior/04-CONTEXT.md` D-01/D-03 |
| Action safety | Every turn executes one of `click`, `type`, `scroll`, `back`; no send action; node ref exists and refreshes before action. | Multi-action batch, unknown action, stale node ref, or action against a non-visible node. | High | `AGENTIC-02`, `AGENTIC-04` |
| Stuck handling | Loop aborts on unchanged tree hash or repeated action/ref, and always respects 5 iterations / 15 seconds. | Loop repeats visible behavior, hangs, or exceeds limits. | Critical | `AGENTIC-03`, context D-06/D-07 |
| Senior communication | Status/failure copy is one warm, brief phrase; contact clarification is one concise question. | Technical error code, long explanation, or ambiguous silence during failure. | High | `UX-02`, `UX-03`, context D-20..D-23 |
| Visual state clarity | Bubble reaches thinking/listening/speaking/error/idle transitions within 200ms and does not add hidden sixth state. | State lags, flickers, or introduces confusing fallback-specific state. | Medium | `OVERLAY-05`, context D-13..D-17 |

### Known Failure Modes in This Domain

- Accessibility trees are noisy and app-specific; node text/contentDescription can be missing or stale.
- WhatsApp UI can change between devices, locales, versions, and install state.
- Touch automation can act on stale nodes unless `nodeInfo.refresh()` is called immediately before `performAction`.
- Older adults may repeat commands if feedback is delayed or unclear; Beto must show/say bounded progress.
- A model may over-generalize from "operate the screen" into trying broader phone control. Phase 4 explicitly forbids this.

### Regulatory / Compliance Context

- No formal medical/financial/legal compliance applies to this hackathon demo.
- AccessibilityService is sensitive platform capability. The implementation must avoid banking/payment flows, avoid hidden background automation, and keep all automation user-initiated or fallback-triggered from an explicit user command.
- Privacy posture carries forward from earlier phases: sanitize cloud-bound text/tree context where applicable and never include sensitive identifiers in logs or prompt traces.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Demo owner / product reviewer | Approves whether fallback feels understandable and trustworthy for an older adult. |
| Android engineer | Verifies Accessibility tree filtering, node refresh, action execution, and abort conditions. |
| UX reviewer | Checks senior typography, contrast, phrase bank, and bubble state clarity on the demo phone. |
| QA runner | Runs the 10-15 reference scenarios and records pass/fail, latency, and failure phrase behavior. |

## 2. Framework Decision

**Selected Framework:** Firebase AI Logic Android SDK + Gemini 2.5 Flash structured output/manual tool loop

**Version:** Firebase Android BoM `34.7.0`; `com.google.firebase:firebase-ai`; model `gemini-2.5-flash`

**Rationale:**
This phase is an Android-client feature, not a server-hosted agent platform. Firebase AI Logic is the only selected stack component already present in the app that lets the team call Gemini from Android without building a backend proxy. It supports Kotlin/Java Android client SDKs, Gemini Developer API vs Vertex AI backend switching, App Check protection, function calling, and structured JSON output. For Phase 4, use Gemini as a single-step decision engine over a compact Accessibility tree. Do not introduce LangGraph, CrewAI, Google ADK, Genkit, or a Python service; those add infra and state machinery that do not fit a 24-36 hour mobile hackathon.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Google ADK | Server/agent-orchestration framework, not needed for one constrained Android client loop; adds deployment and session infrastructure. |
| Genkit | Good Firebase server-side framework, but Phase 4 deliberately avoids backend work. |
| LangGraph | Strong for stateful server workflows, but overkill for a 5-iteration client-side loop and would require Python/TS backend infrastructure. |
| OpenAI Agents SDK | Vendor switch plus backend/proxy problem; no advantage over existing Firebase/Gemini path for this Android demo. |
| Claude / Anthropic Java fallback | Keep as architectural fallback from Phase 3 if Gemini quality fails, but not the Phase 4 default because it requires key/proxy tradeoffs. |
| Direct Gemini REST from Android | Unnecessary; Firebase AI Logic already handles mobile client integration and key protection path. |

**Vendor Lock-In Accepted:** Partial. Phase 4 accepts Gemini/Firebase as the demo path, while preserving the `LlmClient` abstraction from Phase 3 so Anthropic can replace the model if smoke tests fail.

### Entry Point Pattern
```kotlin
@Serializable
data class AgenticDecision(
    val plan: String,
    val action: UiAction,
    val nodeRef: String? = null,
    val text: String? = null,
    val reason: String
)

enum class UiAction { CLICK, TYPE, SCROLL, BACK, ABORT }

class GeminiAgenticDecider(
    private val json: Json = Json { ignoreUnknownKeys = false }
) {
    private val decisionSchema = Schema.obj(
        mapOf(
            "plan" to Schema.string(),
            "action" to Schema.enumeration(listOf("CLICK", "TYPE", "SCROLL", "BACK", "ABORT")),
            "nodeRef" to Schema.string(),
            "text" to Schema.string(),
            "reason" to Schema.string(),
        ),
        optionalProperties = listOf("nodeRef", "text")
    )

    private val model = Firebase.ai(backend = GenerativeBackend.googleAI()).generativeModel(
        modelName = "gemini-2.5-flash",
        generationConfig = generationConfig {
            temperature = 0f
            maxOutputTokens = 512
            responseMimeType = "application/json"
            responseSchema = decisionSchema
        }
    )

    suspend fun decide(tree: String, goal: String, attempt: Int): AgenticDecision {
        val prompt = """
            Sos Beto, un agente de accesibilidad para WhatsApp.
            Objetivo: $goal
            Intento: $attempt de 5
            Reglas:
            - Devolve exactamente una accion ejecutable.
            - Acciones permitidas: CLICK, TYPE, SCROLL, BACK, ABORT.
            - Nunca envies el mensaje.
            - Usa solo refs presentes en el arbol.
            - Si no estas seguro, ABORT.

            Arbol:
            $tree
        """.trimIndent()

        val response = model.generateContent(prompt)
        return json.decodeFromString<AgenticDecision>(response.text.orEmpty())
    }
}
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `GenerativeModel` | Firebase AI Logic model client for Gemini. | Create the `gemini-2.5-flash` decider. |
| `generationConfig` | Model configuration, including temperature, max output tokens, response schema. | Force deterministic structured action output. |
| `responseSchema` | JSON schema that constrains model output. | Ensure every turn returns `plan`, `action`, optional `nodeRef`/`text`, and `reason`. |
| `AgentLoop` | Local Kotlin orchestrator around model calls and Accessibility execution. | Owns iteration/time limits, stuck detection, retries, and final status. |
| `TreeSerializer` | Compact textual representation of visible actionable nodes. | Keeps model context small and references stable via `@e1`, `@e2`. |
| `NodeRefRegistry` | Maps `@eN` refs back to live `AccessibilityNodeInfo`. | Verifies refs exist and refreshes nodes before actions. |

### Common Pitfalls

1. Treating Gemini's output as executable without schema validation. Use strict JSON parsing and abort/retry on malformed output.
2. Sending the full raw Accessibility tree. Filter to visible/actionable/focusable/text nodes and cap at 50 nodes.
3. Executing on stale nodes. Always look up the node by ref and call `refresh()` immediately before `performAction`.
4. Letting the model batch actions. The model may include a short plan, but local code executes exactly one action per iteration.
5. Mixing tool/function calling and structured output without a clear contract. For Phase 4, prefer a structured decision object consumed by local code; the real "tool execution" is Kotlin code, not automatic model function execution.

## 4. Implementation Guidance

**Model Configuration:**
- Model: `gemini-2.5-flash`
- Temperature: `0f`
- Max output tokens: `512` for each agentic decision, while preserving project hard limit `MAX_TOKENS=4000` per turn if a larger tree requires it.
- Response MIME type: `application/json`
- Response schema: `AgenticDecision` with fixed enum action set.
- Backend: `GenerativeBackend.googleAI()` for hackathon free tier; keep switch point for Vertex AI if quota/stability requires it.

**Core Pattern:**
The model is a decider, not an autonomous executor. Local Kotlin owns state, guardrails, and execution:

1. `IntentBranch`/dispatcher reports clear WhatsApp failure.
2. Beto speaks: "Dame un segundo, lo intento de otra forma."
3. `AgentLoop` sets bubble to thinking and starts timer/iteration count.
4. `BetoAccessibilityService` provides filtered tree + registry.
5. Gemini returns `AgenticDecision(plan, action, nodeRef?, text?, reason)`.
6. Local code validates schema, allowed action, app scope, ref existence, and repeated-action/tree-hash rules.
7. Execute exactly one action.
8. Stop on success, abort, stuck, 5 iterations, 15 seconds, or 3 failed contact clarification/search attempts.

**Tool Use:**
- Do not expose broad Android operations as model tools.
- The allowed action enum is exactly: `CLICK`, `TYPE`, `SCROLL`, `BACK`, `ABORT`.
- `TYPE` requires a `nodeRef` pointing to an editable/focusable node and `text`.
- `CLICK`/`SCROLL` require valid node refs unless using a controlled global back action for `BACK`.
- The model must never receive or produce a `SEND` action.

**State Management:**
- Keep state in memory inside `AgentLoop`: `startedAt`, `iteration`, `lastTreeHash`, `lastActionSignature`, `contactClarificationAttempts`, `goal`, `message`.
- Do not persist learned aliases in Phase 4.
- Emit typed `AgentEvent`s for `AgenticStarted`, `AgenticIteration`, `AgenticSucceeded`, `AgenticAborted`, and bubble state changes.

**Context Window Strategy:**
- Serialize at most 50 nodes.
- Include only: ref, text/contentDescription/resourceId if useful, className simplified, clickable/scrollable/focusable/editable booleans, bounds bucket if needed.
- Exclude sensitive raw data where not needed; redact phone/DNI/card-like strings before cloud prompt/logging.
- Include only the current goal, iteration number, previous action signature, and current tree.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with Pydantic

Pydantic is not an Android runtime dependency here. Use this schema as the canonical eval/test contract and mirror it in Kotlin with `@Serializable` plus Firebase `Schema.obj(...)`.

```python
from enum import Enum
from pydantic import BaseModel, Field, model_validator

class UiAction(str, Enum):
    CLICK = "CLICK"
    TYPE = "TYPE"
    SCROLL = "SCROLL"
    BACK = "BACK"
    ABORT = "ABORT"

class AgenticDecision(BaseModel):
    plan: str = Field(min_length=1, max_length=240)
    action: UiAction
    nodeRef: str | None = Field(default=None, pattern=r"^@e[0-9]+$")
    text: str | None = Field(default=None, max_length=280)
    reason: str = Field(min_length=1, max_length=200)

    @model_validator(mode="after")
    def validate_action_requirements(self):
        if self.action in {UiAction.CLICK, UiAction.SCROLL, UiAction.TYPE} and not self.nodeRef:
            raise ValueError("nodeRef required for node actions")
        if self.action == UiAction.TYPE and not self.text:
            raise ValueError("text required for TYPE")
        if self.action != UiAction.TYPE and self.text:
            raise ValueError("text only allowed with TYPE")
        return self
```

Runtime behavior:
- First malformed response: one repair retry with the same tree and stricter "return valid JSON only" instruction.
- Second malformed response: abort warmly, log `Beto-LLM` and `Beto-Action`, return bubble to idle/error sequence per UX plan.
- Never coerce unknown actions into allowed actions.

### Async-First Design

Use coroutines. `AgentLoop.run(...)` should be `suspend` and enforce timeout with `withTimeout(15.seconds)`. All model calls and Accessibility operations should be serialized through one loop job so two fallbacks cannot run at once. UI/bubble updates happen on Main; tree serialization/model calls can run off Main.

### Prompt Engineering Discipline

- System/developer rules are static in `AgenticPrompts.kt`; dynamic data is only goal, attempt count, previous action, and tree.
- Prompt language is Spanish because tool names/descriptions and user command are Spanish, but enum/action names remain English constants.
- Few-shots should include only the controlled WhatsApp scenario and abort behavior. Avoid teaching broad phone control.
- Include hard prohibitions in every prompt: no send, no calls, no SMS/Maps, no non-WhatsApp apps, one action only.

### Context Window Management

Accessibility trees can explode. The serializer is the context manager:
- Filter to visible nodes with at least one useful property: clickable, longClickable, focusable, editable, scrollable, text, or contentDescription.
- Cap to 50 nodes and prefer nodes whose text/class suggests WhatsApp contact/search/message composition.
- Stable refs are per-iteration only. Never reuse a ref across a fresh tree without registry lookup.

### Cost and Latency Budget

- Target one model call per iteration, maximum 5.
- For demo, expected path should resolve in 1-3 iterations.
- Cache is not useful for raw Accessibility trees; instead cache only prompt constants/model instance.
- Abort early on stuck signals rather than spending all 5 calls.
- Log per-iteration latency under `Beto-LLM` and total loop duration under `Beto-Action`.

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Scope allow-list | Current package is not WhatsApp or fallback was not invoked from WhatsApp failure path. | Abort, speak warm failure if user-facing, return idle. |
| Action allow-list | Model action not in `CLICK`, `TYPE`, `SCROLL`, `BACK`, `ABORT`. | Reject; one repair retry; abort if repeated. |
| Send prevention | Node text/contentDescription/action suggests "Send", "Enviar", send button, or outgoing message send. | Block action and abort; never click send. |
| Node ref validation | `nodeRef` missing, unknown, not visible, or `refresh()` fails. | Reject action; retry once or abort depending on iteration. |
| Stuck detector | Same tree hash unchanged or same action/ref repeated twice. | Abort warmly and return bubble to idle. |
| Loop budget | Iteration > 5 or elapsed > 15 seconds. | Abort warmly and return bubble to idle. |
| Contact attempts | Contact clarification/search attempts > 3. | Leave WhatsApp as-is, speak warm failure, return idle. |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| Agentic success rate | Every manual demo run. | Adjust prompt/tree filtering or narrow scenario. |
| Malformed output rate | Every model call in testing. | Tighten schema/prompt; consider fallback provider if persistent. |
| Abort reason distribution | Every failed fallback run. | Fix dominant abort cause first. |
| Average loop duration | Every fallback run. | Reduce tree size or abort earlier if >10s average. |
| UX phrase compliance | Every phrase-bank addition. | Replace freeform copy with approved phrase. |

## 7. Production Monitoring

**Tracing Tool:** Logcat/Timber for hackathon. If post-hackathon tracing is needed, adopt Arize Phoenix or Langfuse behind a backend/proxy; do not add hosted tracing to the client-only demo path.

**Key Metrics to Track:**
- `agentic_started_count`
- `agentic_success_count`
- `agentic_abort_reason`
- `agentic_iterations`
- `agentic_elapsed_ms`
- `agentic_malformed_response_count`
- `bubble_state_transition_ms`

**Alert Thresholds:**
- Any send action attempted: blocker.
- Any loop exceeds 15 seconds: blocker.
- Any fallback starts without explicit WhatsApp failure: blocker.
- More than 20% malformed outputs in smoke test: consider Anthropic fallback path from `LlmClient`.
- Bubble transition over 200ms: fix before demo.

**Smart Sampling Strategy:**
For the hackathon, review 100% of fallback runs. Prioritize runs with retries, aborts, malformed responses, repeated actions, and total duration over 8 seconds.
