# Plan 04-02: WhatsApp-Only Agentic Fallback Loop Integration

<objective>
Wire the constrained agentic primitives into a real Phase 4 fallback path: when WhatsApp Intent launch clearly fails, Beto speaks the fixed warm phrase, enters thinking state, reads the Accessibility tree, asks Gemini for one structured decision per iteration, executes one safe action through `BetoAccessibilityService`, and stops on success, limits, stuck detection, or 3 failed contact clarification/search attempts.
</objective>

<user_story>
As an older adult, if Beto cannot open WhatsApp through the fixed path, I want it to try one simple alternate path without making me understand the failure, and stop calmly if it cannot find the contact.
</user_story>

<must_haves>
- D-01: Fallback trigger is narrow and clear-failure-biased.
- D-02: Before fallback, Beto says exactly `Dame un segundo, lo intento de otra forma.`
- D-03: Phase 4 fallback applies only to WhatsApp.
- D-04: Success means the message is visible/filled in the WhatsApp compose field; Beto never auto-sends.
- D-05: Allowed actions are click, type, scroll, back, and abort.
- D-06: Loop respects 5 iterations, 15 seconds, and 4000 tokens per turn.
- D-07: Abort on unchanged tree hash or repeated action/ref.
- D-08: Gemini can return a short internal plan, but local code executes exactly one action per iteration.
- D-09: Controlled scenario is WhatsApp contact/search fallback for `mi nieto`.
- D-10: Beto asks how the contact is named/saved and retries at most 3 times.
- D-11: After 3 failed attempts, Beto leaves WhatsApp as-is, speaks warm failure, and returns bubble to idle.
- D-12: No alias/contact learning persists.
- AI-SPEC: Use Firebase AI Logic / Gemini 2.5 Flash structured JSON output and local Kotlin execution.
- UI-SPEC: Fallback uses thinking state and one-sentence failure copy.
</must_haves>

<threat_model>
## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Fallback runs when fixed Intent succeeded | high | Trigger only from explicit `ActionResult.Failed` or test-only injected failure path. |
| Model clicks WhatsApp send button | critical | No `SEND` action exists; service blocks nodes whose text/contentDescription suggests send/enviar. |
| Model operates outside WhatsApp | high | Check foreground/package context before each loop iteration; abort if not WhatsApp. |
| Loop hangs or repeats visible behavior | high | Enforce `withTimeout(15000)`, max 5 iterations, and `StuckDetector`. |
| Warm UX violated by technical failures | medium | All user-facing fallback/failure phrases come from `PhraseBank`; technical reasons stay in logs only. |
</threat_model>

    goal: String,
    contactHint: String,

Concrete behavior:

- Emit `BubbleStateChanged("thinking")` and `AgenticStarted(goal)`.
- Use `withTimeout(AgenticLimits.TIMEOUT_MS)`.
- Iterate from 1 through `AgenticLimits.MAX_ITERATIONS`.
- On each iteration, request a new snapshot from `BetoAccessibilityService`.
- Abort if snapshot is null or node count is 0.
- Call `GeminiAgenticDecider.decide(...)`.
- Validate the decision.
- Check `StuckDetector` before executing action.
- Execute exactly one action via `performAgenticAction`.
- Emit `AgenticIteration` with iteration/action/nodeRef/treeHash.
- Track contact clarification/search attempts; after 3 failed attempts, return failure phrase and idle state.
- Success condition is field-filled/prepared state, not sent message. Implement the safest available check from current tree text/editable field; if uncertain, keep looping until limits rather than claiming sent.
- On terminal success, emit `AgenticSucceeded`, speak `PhraseBank.messagePreparedSuccess`, and return idle.
- On terminal failure, emit `AgenticAborted`, speak `PhraseBank.contactNotFoundAfterAttempts` or generic warm failure, and return idle.
